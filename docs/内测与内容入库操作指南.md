# 门店专属 Hermes —— 内测与内容入库操作指南

> 目的：解决三件事：手机真实登录内测、真实门店知识库上传、销售底层方法论/行业标杆案例持续入库。

---

## 1. 手机端真实登录内测

### 1.1 同一 Wi-Fi 下本地内测

适合：你人在店里或办公室，手机和 Mac 连同一个 Wi-Fi。

1. 在 Mac 项目目录启动服务（已内置 `dev:lan` 脚本，自动绑定 0.0.0.0:3000）：

```bash
cd /Users/wangkai/store-ai-assistant
npm run dev:lan
```

（等价于 `next dev -H 0.0.0.0 -p 3000`；仍可用旧写法 `npm run dev -- -H 0.0.0.0 -p 3000`。）

2. 查看 Mac 局域网 IP：

```bash
ipconfig getifaddr en0
```

如果输出是 `192.168.1.23`，手机浏览器打开：

```text
http://192.168.1.23:3000/start
```

3. 用真实账号登录，不要只用一键演示登录。

4. iPhone 可以点浏览器分享按钮，选择「添加到主屏幕」，之后像 App 一样打开。

### 1.2 常见打不开原因

| 现象 | 原因 | 处理 |
|---|---|---|
| 手机打不开 `localhost:3000` | 手机上的 localhost 指手机自己，不是 Mac | 必须用 Mac 的局域网 IP |
| 手机打不开 IP 地址 | dev 服务只绑定本机 | 用 `npm run dev -- -H 0.0.0.0 -p 3000` |
| 手机和电脑互相访问不了 | 不在同一 Wi-Fi / Wi-Fi 开了访客隔离 | 换同一网络或关闭 AP 隔离 |
| 登录后状态异常 | Supabase/环境变量异常 | 先在 Mac 打开 `/start` 看系统状态 |

### 1.3 外网真实内测

如果你要让不在同一 Wi-Fi 的门店员工测试，本地 IP 不够，需要二选一：

- 临时内测：用 Tailscale / Cloudflare Tunnel / ngrok 暴露本地服务。
- 稳定内测：部署到 Vercel 或云服务器，绑定一个测试域名。

正式高密度内测建议用「稳定内测」方式，避免手机网络、Wi-Fi、Mac 睡眠导致测试中断。

## 2. 真实门店知识库怎么上传

### 2.1 入口

老板/店长登录后：

```text
/admin/knowledge/upload
```

当前支持：

- `.md`
- `.txt`
- `.docx`
- `.pdf`

上传时需要填写：

- 资料标题
- 资料分类
- 可见角色
- 标签
- 是否启用
- 备注说明

上传后系统会：

1. 提取文本；
2. 按标题/段落切成知识片段；
3. 写入 `knowledge_documents` 和 `knowledge_chunks`；
4. 如果 Qwen embedding key 可用，会同步生成向量；失败不阻断上传，可后续跑回填脚本。

### 2.2 不建议怎么上传

不要把所有真实资料打包成一个几十页大文档直接上传。这样 AI 检索会变钝，员工问具体问题时容易混用上下文。

不建议：

- 一个文档里同时放活动、项目、价格、客诉、员工制度；
- 文件标题很泛，比如「门店资料」「知识库」「培训资料」；
- 缺少适用角色；
- 话术里包含绝对疗效、根治、一次见效、无风险等高风险表达。

### 2.3 推荐拆分方式

每份文档只解决一类问题。

| 内容类型 | 推荐分类 | 推荐文件名 |
|---|---|---|
| 门店定位/品牌故事 | 品牌介绍 / 门店定位 | `门店定位与品牌介绍.md` |
| 项目功效、适合人群、流程 | 项目介绍 / 护理流程 | `项目-小气泡清洁.md` |
| 价格、套餐、权益边界 | 价格表 | `价格表-2026年6月.md` |
| 当月活动 | 活动方案 | `活动-2026年6月补水季.md` |
| 员工销售话术 | 销售话术 / 客户异议处理 | `话术-客户嫌贵与考虑一下.md` |
| 售后/回访标准 | 售后SOP / 服务流程 | `SOP-体验后1-3-7天回访.md` |
| 客诉/异常处理 | 客诉处理 / 合规规则 | `SOP-皮肤异常升级处理.md` |
| 门店优秀成交案例 | 优秀案例 | `案例-新客体验转卡.md` |
| 朋友圈/小红书素材 | 朋友圈文案 / 小红书文案 | `文案-补水项目朋友圈.md` |

### 2.4 建议每份知识库文档固定结构

见模板：

```text
docs/templates/门店知识库文档模板.md
```

## 3. 内置销售底层内容怎么新增

### 3.1 先区分两类内容

| 类型 | 应该放哪里 | 用途 |
|---|---|---|
| 这家店的事实资料 | 普通知识库 `knowledge_documents / knowledge_chunks` | 回答本店活动、项目、价格、流程、制度 |
| 销售底层方法论/行业标杆打法 | `growth_playbooks` 方法论库 | 帮 AI 判断场景、客户心理、成交策略、话术、下一步动作 |

一句话：  
门店事实用「知识库上传」；销售方法论和标杆案例用「方法论库」。

### 3.2 当前系统的方式

当前 `growth_playbooks` 主要通过脚本维护：

```text
scripts/seed-playbooks.mjs
```

这个脚本现在是系统级方法论，`store_id=null`，全门店通用。

执行方式：

```bash
cd /Users/wangkai/store-ai-assistant
node --env-file=.env.local scripts/seed-playbooks.mjs
npx tsx --env-file=.env.local scripts/backfill-embeddings.ts
```

注意：现有脚本会先清空系统级 `store_id is null` 的 playbooks 再写入，所以后续扩充时要把旧内容和新内容一起保留在脚本里。

### 3.3 用 CSV 导入脚本持续新增（已实现，不依赖后台）

已新增轻量脚本 `scripts/import-playbooks-from-csv.ts`，你以后只要维护 CSV，不必改代码。

读取模板：

```text
docs/templates/growth-playbooks-template.csv
```

导入命令：

```bash
cd /Users/wangkai/store-ai-assistant

# 系统级（store_id=null，全门店通用，默认）
npx tsx --env-file=.env.local scripts/import-playbooks-from-csv.ts

# 门店级（仅某门店可见）
npx tsx --env-file=.env.local scripts/import-playbooks-from-csv.ts --store=<storeId>

# 指定其它 CSV 文件
npx tsx --env-file=.env.local scripts/import-playbooks-from-csv.ts --file=docs/templates/xxx.csv

# 导入后务必生成向量，否则新方法论无法被语义检索命中
npx tsx --env-file=.env.local scripts/backfill-embeddings.ts
```

**注意事项：**

- **幂等**：按 `(scenario_key + store_id)` 去重，已存在则更新、不存在则新增。重复跑安全。
- **与 `seed-playbooks.mjs` 的区别**：seed 脚本会先清空系统级旧数据再写入；本导入脚本**不清空、只增量 upsert**，适合持续扩充，更安全。
- **必跑回填**：导入后没生成向量前，新方法论只能靠 bigram 关键词命中，语义检索召不到。
- **`source` / `tags` 两列**：仅作内容管理元信息，`growth_playbooks` 表无对应列，导入时自动跳过（不报错）。
- **多值字段**：`applicable_roles` / `applicable_stages` 用逗号分隔（如 `consultant,reception`）；若字段内容本身含逗号，整列要用英文双引号包裹（CSV 规则）。
- **必填**：`scenario_key`、`module`、`title` 缺任一则跳过该行。

导入字段说明：

| 字段 | 含义 |
|---|---|
| `scenario_key` | 唯一场景 key，如 `price_objection_high_end` |
| `module` | 模块，如 销售成交 / 客户分型 / 老客复购 |
| `title` | 标题 |
| `scene` | 适用场景 |
| `customer_psychology` | 客户心理 |
| `common_mistakes` | 员工常犯错误 |
| `strategy` | 推荐策略 |
| `scripts` | 可用话术 |
| `follow_up_questions` | 追问问题 |
| `next_action` | 下一步动作 |
| `risk_note` | 合规风险 |
| `applicable_roles` | 适用角色，逗号分隔 |
| `applicable_stages` | 客户阶段，逗号分隔 |
| `source` | 来源，如 内部沉淀 / 标杆案例 / 公开资料（**不入库，仅 CSV 内备注**） |
| `tags` | 标签（**不入库，仅 CSV 内备注**） |

### 4.3 行业标杆案例库

后续可以把行业标杆做成「案例型 playbook」，不是普通文章。

每个标杆案例建议固定写成：

- 背景：这是什么类型门店；
- 场景：遇到什么经营问题；
- 客户类型：对应什么客户；
- 原错误做法：过去怎么做不对；
- 标杆做法：高手怎么做；
- 可复制动作：普通员工今天能做什么；
- 可用话术：怎么开口；
- 指标结果：提升了什么；
- 注意边界：哪些话不能说、哪些动作不能做；
- 适用门店：哪些门店适合，哪些不适合。

我可以帮你找和整理：

- 连锁品牌服务 SOP 的公开信息；
- 大众点评/小红书/团购平台上的用户反馈模式；
- 常见客诉和差评原因；
- 活动转化、老客复购、新客体验的通用打法；

如果是非公开标杆客户案例，需要你提供原始材料，并做脱敏处理。

## 5. 现在建议的执行顺序

1. 先把手机端真实登录跑通。
2. 用 3-5 份真实知识库小文档测试，不要一次性全量上传。
3. 选 10 个真实门店场景问 AI：活动、价格、项目、回访、客诉、异常、成交、复购、老板经营、员工培训。
4. 记录哪些回答有用、哪些不准、哪些缺资料。
5. 再扩充销售底层方法论和标杆案例库。
6. 最后才考虑新增复杂后台入口。

当前阶段的重点不是“资料越多越好”，而是先验证：真实资料进来后，员工问得出、AI 找得准、话术能用、老板能看出经营动作。
