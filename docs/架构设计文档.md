## 一、项目概述

**门店 AI 经营助手** 是面向美容院 / 皮肤管理 / SPA / 轻医美门店的私有化 AI 经营辅助系统。它不是通用聊天机器人，而是一套集 **门店经验沉淀 + 员工工作指导 + 老板管理复盘 + 风险控制** 于一体的 SaaS 平台。

### 用户角色

| 角色 | 标识 | 核心场景 |
|------|------|----------|
| 老板 | `owner` | 经营驾驶舱、日报、员工管理、权限配置 |
| 店长 | `manager` | 全店管理、任务分派、风险处理、客户复盘 |
| 咨询师 | `consultant` | AI 话术辅助、客户沟通、成交跟进 |
| 美容师 | `beautician` | SOP 查询、服务执行引导、客户反馈 |
| 前台 | `receptionist` | 客户接待、预约管理、基础问答 |
| 运营 | `operator` | 活动文案、通知发布、客户唤醒 |

## 五、核心模块设计

### 5.1 AI 问答管线 (pipeline.ts)

这是整个系统最核心的业务流程，`answerQuestion()` 函数串联了完整的问答链路：

```
用户提问
  │
  ▼
┌─────────────────┐
│ 1. 问题分类      │ ← classifyQuestion() 规则匹配 12 种分类
│    初判风险      │   关键词规则判定 L1-L4
└───────┬─────────┘
        ▼
┌─────────────────┐
│ 2. 多维检索      │
│  ├ 知识库向量检索│ ← pgvector match_knowledge_chunks (语义)
│  │   (bigram兜底)│   失败回退中文 bigram 关键词匹配
│  ├ 方法论检索    │ ← playbooks.matchByEmbedding (向量)
│  ├ 客户画像      │ ← customers 表
│  ├ 门店记忆      │ ← store-memory 提炼经验
│  └ 业务上下文    │ ← 排班/通知/活动/项目 (动态注入)
└───────┬─────────┘
        ▼
┌─────────────────┐
│ 3. 类型判定      │ ← 知识库有答案 → knowledge
│    风险定级      │   通用建议 → general
│                  │   不确定需确认 → need_confirm
│                  │   高风险 → risk
└───────┬─────────┘
        ▼
┌─────────────────┐
│ 4. 构建 Prompt   │ ← 角色化 System Prompt (6 种角色不同视角)
│                  │   + 合规约束 + 方法论融合
│                  │   + 结构化输出要求 (9 段式)
└───────┬─────────┘
        ▼
┌─────────────────┐
│ 5. LLM 调用      │ ← adapter.ts → mock/deepseek/qwen
│   (自动降级)     │   失败自动回退 mock 兜底
└───────┬─────────┘
        ▼
┌─────────────────┐
│ 6. 后处理        │
│  ├ 禁用词检查    │ ← 内置 15 个 + 门店自定义禁用词
│  ├ 合规清洗      │ ← 保守化替换承诺词，剔除绝对化句式
│  └ 风险升级      │ ← 高风险自动进 risk_logs
└───────┬─────────┘
        ▼
┌─────────────────┐
│ 7. 落库          │ ← chat_messages (回答 + 风险等级 + 检索来源)
│    更新互动时间线 │
└───────┬─────────┘
        ▼
┌─────────────────┐ ─ ─ ─ ─ 旁路处理 (异常静默)
│ 8. 副作用处理    │
│  ├ 记忆沉淀      │ ← 抽取客户事实 → memory_items
│  ├ 增长机会      │ ← upsert opportunities
│  ├ 风险记录      │ ← L3/L4 风险入库
│  ├ 待确认问题    │ ← need_confirm 入库
│  └ 知识库缺口    │ ← 无答案匹配入库
└─────────────────┘
```

### 5.2 知识库系统

```
文件上传
  │
  ├── parse.ts    支持格式: md/txt/docx/pdf/xlsx/csv/pptx/图片
  │               图片 → 千问 Vision OCR
  │
  ├── chunk.ts    按 Markdown 标题分节 + 段落合并
  │               每段 500-900 字，保持语义完整
  │
  ├── embedding.ts 千问 text-embedding-v3 (1024维)
  │               支持批量 + 单条查询向量化
  │
  └── retrieve.ts  向量语义召回 (pgvector)
                  失败回退 bigram 关键词匹配
                  严格隔离: store_id + status + visible_roles
```

### 5.3 客户经营系统

```
客户池自动分池 (customer-pools.ts)
  ├── 今日到店池   今日有预约/互动
  ├── 新客池       近 7 天新录入
  ├── 新成交池     近 30 天成交
  ├── 老客池       正常活跃客户
  ├── 沉睡池       超 30 天无互动
  └── 风险池       流失风险 / 投诉风险

长记忆系统 (memory-extractor.ts)
  从问答中自动抽取客户事实
  范围: customer / employee / store 三级
  可信度标签 + 画像回写

增长机会追踪 (opportunity.ts)
  8 种机会类型，upsert 去重
  逾期自动判定 + 紧急度排序
  AI Copilot 每日优先建议
```

## 六、AI 能力层

### 6.1 AI Provider 适配

```
                     AI_PROVIDER 环境变量
                            │
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
        mock            deepseek          qwen
    (结构化兜底)    (OpenAI 兼容)    (文本+Vision+ASR)
                            │
                    失败自动降级到 mock
```

### 6.2 多模态能力

| 能力 | 模型 | 场景 |
|------|------|------|
| 文本对话 | DeepSeek / Qwen 文本模型 | 核心问答、角色化回答 |
| 图片识别 | Qwen Vision | 客户聊天截图、活动海报、客户反馈图片、美团点评截图 |
| 语音转写 | 阿里云通义 Paraformer | 会谈录音转写 + 说话人分离 (异步) |
| 向量嵌入 | Qwen text-embedding-v3 | 知识库语义检索、方法论匹配 (1024维) |

### 6.3 Prompt 工程

- **角色化 Prompt**：6 种角色有完全不同的回答视角和侧重点
- **9 段式结构化输出**：客户判断 / 沟通策略 / 建议话术 / 追问问题 / 下一步动作 / 风险提醒 / 是否升级 / 是否补充标签 / 参考知识来源
- **合规约束**：疗效承诺禁止、绝对化表述剔除、禁用词过滤
- **方法论融合**：系统增长方法论 (playbooks) 融入 Prompt 上下文

## 七、权限与安全

### 7.1 认证流程

```
浏览器                          Next.js                     Supabase
  │                               │                            │
  │  POST /api/login              │                            │
  │  {email, password} ──────────►│  signInWithPassword ──────►│
  │                               │◄───── session + cookies ──│
  │◄────────── set-cookie ───────│                            │
  │                               │                            │
  │  Middleware (每次请求)          │                            │
  │                               │  getSession() ────────────►│
  │                               │◄───── user ───────────────│
  │                               │  未登录 → 重定向 /login      │
```

### 7.2 权限矩阵

| 模块 | 操作 | owner | manager | consultant | beautician | receptionist | operator |
|------|------|-------|---------|------------|------------|-------------|----------|
| 员工管理 | 查看/编辑 | store | store | - | - | - | - |
| 知识库 | 管理 | store | store | - | - | - | - |
| 知识库 | 查看 | store | store | self | self | self | self |
| 客户 | 查看 | store | store | self | self | - | - |
| 风险处理 | 处理 | store | store | - | - | - | - |
| 通知发布 | 创建 | store | store | - | - | - | store |
| 经营报告 | 查看 | store | store | - | - | - | - |
| 配置管理 | 编辑 | store | store | - | - | - | - |

### 7.3 数据隔离

- **数据库层**：RLS 开启，仅 service_role 可绕过
- **应用层**：所有查询严格带 `store_id` 过滤
- **知识隔离**：同门店 + 角色可见 (`visible_roles`) + 启用状态
- **客户隔离**：`data_scope = self` 时仅看自己负责的客户

### 7.4 风险控制机制

| 风险等级 | 说明 | 处理方式 |
|----------|------|----------|
| L1 信息类 | 正常问题 | 直接回答 |
| L2 注意类 | 价格/退款/活动叠加 | 自动转"待确认" |
| L3 警告类 | 皮肤异常/效果承诺 | 记录风险 + 升级通知 |
| L4 严重类 | 医疗建议/投诉/过敏 | 自动拦截 + 风险记录 + 升级 |

## 九、可替换适配层设计

系统核心设计原则：**页面和组件不直接接触 Supabase**，所有外部依赖收敛在 4 个适配层。

| 适配层 | 职责 | 迁移方案 |
|--------|------|----------|
| `lib/auth/` | 登录态、账号创建、浏览器登出 | 替换 `provider.ts`，保持 `getAuthContext()` 签名不变 |
| `lib/db/index.ts` | 30+ 语义化数据访问模块 | 新增同接口实现 (如 `lib/db/mysql.ts`)，换 import 即可 |
| `lib/storage/` | 知识库原始文件存储 | 实现 `saveOriginal()` 接口，默认 `none` 不依赖桶 |
| `lib/ai/adapter.ts` | AI 模型调用 + 问答流程 | `AI_PROVIDER` 切换 或 新增厂商适配 |

验证：`grep -rl "@supabase" app components lib` 仅命中 `lib/auth`、`lib/db`、`lib/storage`、`lib/supabase`。

## 十二、设计原则总结

1. **分层解耦**：supabase → db → auth/ai → actions → 页面，每层独立可替换
2. **适配器模式**：AI Provider、存储 Provider、数据库均可切换，只需改对应适配层
3. **多级降级**：向量检索失败回退 bigram；LLM 调用失败回退 mock；存储关闭回退 null
4. **旁路设计**：记忆沉淀、增长机会抽取均为旁路处理，异常静默吞掉，不影响主回答
5. **门店隔离**：所有数据操作严格按 `store_id` + `visible_roles` + `status` 隔离
6. **合规红线**：禁用词检查、疗效承诺过滤、风险升级机制贯穿全链路
7. **移动端优先**：max-w-md 布局 + PWA standalone + 安全区适配
8. **渐进增强**：mock 模式下无外部依赖也可跑通全流程，逐步接入真实 AI
