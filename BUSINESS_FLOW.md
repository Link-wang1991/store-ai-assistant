# 门店 AI 助手 · 业务流程文档

## 📋 系统概览

**门店 AI 经营助手** 是一套私有化的、基于门店自有知识库的 AI 工作指导系统，覆盖以下核心流程：

1. **员工工作指导**：员工通过 AI 对话获取工作建议（需求响应阶段）
2. **经营数据复盘**：老板/店长在后台查看员工提问趋势、风险预警、日报统计（洞察阶段）
3. **知识库管理**：老板/店长上传门店 SOP、话术、规则等，AI 据此指导员工（供给阶段）
4. **任务流转**：老板分配任务给员工（执行阶段）
5. **风险管理**：系统自动检测违规表达、不当承诺等，升级高风险提问给老板确认

## 🎭 角色体系

### 6 个核心角色

| 角色 | 中文名 | 主要职责 | 进入的页面 |
|------|--------|--------|----------|
| **owner** | 老板 | 门店经营决策者、知识库建设者、风险最终负责人 | `/admin`（经营驾驶舱） |
| **manager** | 店长 | 门店日常管理、员工协调、客户跟进指导 | `/admin`（受限版后台） |
| **consultant** | 咨询师 | 新客咨询、销售转化、客户跟进 | `/work`（工作台） + `/chat`（AI 对话） |
| **beautician** | 美容师 | 项目施术、服务标准化、客户体验 | `/work`（工作台） + `/chat`（AI 对话） |
| **receptionist** | 前台 | 客户接待、排期预约、电话/微信回复 | `/work`（工作台） + `/chat`（AI 对话） |
| **operator** | 运营 | 内容运营、新媒体、活动策划、文案 | `/work`（工作台） + `/chat`（AI 对话） |

**权限梯度**：
- **管理员**（老板/店长）：可进入 `/admin` 后台，管理员工、知识库、任务、风险、报告
- **普通员工**（咨询师/美容师/前台/运营）：只能看 `/work` 工作台，无法进入管理后台

## 🔄 核心业务流程（按角色）

### 1️⃣ **咨询师（Consultant）工作流**

#### 场景：新客咨询、销售跟进

```
客户咨询
  ↓
咨询师在 /chat 提问 AI
  ↓
AI 响应流程：
  ├─ 1. 检索知识库（项目介绍、价格表、销售话术）
  ├─ 2. 获取客户档案（消费能力、决策风格、当前阶段）
  ├─ 3. 生成话术建议（基于客户档案和知识库）
  ├─ 4. 检测禁用词/风险
  └─ 5. 返回回答 + 风险评级
  ↓
咨询师采纳建议，继续跟进客户
```

**快捷问题模板**：
- "客户嫌贵怎么回？"
- "客户说考虑一下怎么回？"
- "客户对比别家怎么回？"
- "帮我介绍某个项目"
- "帮我判断这个客户适合哪个项目"

**对应知识库分类**：
- 项目介绍
- 价格表
- 销售话术
- 客户异议处理
- 优秀案例

### 5️⃣ **店长（Manager）工作流**

#### 场景 A：员工培训与指导
```
每日晨会/周会
  ↓
店长在 /admin → 报告 查看昨日/本周员工提问汇总
  ↓
AI 分析报告包括：
  ├─ 员工提问 TOP 10 问题
  ├─ 频繁提问说明的知识库缺口
  ├─ 每个员工的能力短板
  └─ 风险预警（禁用词、医美合规、不当承诺）
  ↓
店长针对性培训或补充知识库内容
```

#### 场景 B：任务分配与跟进
```
店长在 /admin → 任务 创建任务
  ↓
选择任务类型：
  ├─ 客户跟进（分配给咨询师）
  ├─ 活动执行（分配给运营/咨询师）
  ├─ 员工培训（分配给全员）
  ├─ 老客唤醒（分配给咨询师）
  ├─ 朋友圈发布（分配给运营）
  ├─ 客诉处理（分配给责任人）
  ├─ 服务复盘（分配给项目负责人）
  └─ 知识库补充（分配给运营/老板）
  ↓
员工在 /work 看到任务，逐步完成（待开始 → 进行中 → 已完成）
  ↓
店长在 /admin 跟踪任务进度、给反馈
```

#### 场景 C：知识库管理
```
店长在 /admin → 知识库 上传/编辑资料
  ↓
上传内容包括：
  ├─ 新产品介绍（含施术流程、禁忌）
  ├─ 销售话术（按客户类型/异议拆分）
  ├─ 新活动方案（含目标、执行节点、员工任务拆解）
  ├─ 员工岗位手册（职责、标准动作、评价指标）
  ├─ SOP 文档（护理流程、应急预案）
  ├─ 禁用词补充（防止违规表达）
  └─ 优秀案例和改进点
  ↓
AI 自动分类、切分、生成向量化嵌入
  ↓
员工提问时即可调用（无需手动改进质量，AI 自动匹配度检索）
```

     ├─ 有异议的员工提问
     └─ 逾期未完成的任务

在 /admin → 员工管理 查看每个员工的档案

  ├─ 提问频次（高频说明知识掌握度不足）
  ├─ 高风险提问占比（反映合规意识）

#### 场景 C：知识库建设（长期）
```
定期审视 /admin → 知识库 → 缺口分析
  ↓
AI 自动检测：
  ├─ 员工频繁提问但知识库没有的话题（缺口）
  ├─ 高风险提问反映的 SOP 不清（改进点）
  ├─ 每周新添加的知识库资料（建设进度）
  └─ 禁用词命中率（合规改进）
  ↓

  ├─ 补充缺口资料（邀请产品团队、医学顾问）
  ├─ 优化高风险问题的标准答案
  ├─ 定期更新禁用词列表
  └─ 制定每周知识库增长目标（如每周新增 5-10 份资料）
```

  ├─ 哪些角色能上传知识库（默认只有老板/店长）
  ├─ 哪些角色能看客户数据、销售数据（按职位需要）
  ├─ 哪些角色能审核/确认高风险提问
  └─ 哪些角色能导出报告、财务数据

## 🤖 AI 问答的核心流程

```
员工提问
  ↓
【第 1 步】问题分类
  ├─ 自动判断问题类型（项目介绍、销售话术、客诉处理 etc.）
  └─ 推断提问人角色（咨询师、美容师、前台等）
  ↓
【第 2 步】知识库检索
  ├─ 将问题向量化
  ├─ 在门店知识库中做 Top-5 相似性检索
  ├─ 返回最匹配的文档片段（chunks）
  └─ 匹配度评分（置信度）
  ↓
【第 3 步】客户档案上下文（可选）
  ├─ 如果提问涉及具体客户
  ├─ 获取该客户的档案、消费能力、决策风格、近期互动
  └─ 融入 AI 提示词
  ↓
【第 4 步】AI 生成回答
  ├─ System Prompt：结合角色职责、知识库内容、门店上下文
  ├─ User Input：员工提问 + 检索到的相关资料 + 客户档案（如有）
  ├─ LLM 调用（支持 mock / deepseek / openai / claude，可在 .env.local 切换）
  └─ 返回自然语言回答
  ↓
【第 5 步】风险检测
  ├─ 违规词检测：回答中是否出现禁用词（根、治、永久、保证等）
  ├─ 医美合规检测：是否有不当医学声称（AI 自动分类为 L4 高风险）
  ├─ 承诺度检测：是否过度承诺效果
  └─ 评级结果：L1 (知识库明确答案) / L2 (通用建议) / L3 (需确认) / L4 (高风险)
  ↓
【第 6 步】回答返回给员工
  ├─ 若无风险 (L1/L2)：直接展示回答，可使用
  ├─ 若需确认 (L3)：展示回答 + 风险标签，可用但标记为"已确认"
  ├─ 若高风险 (L4)：回答被阻止，转发给老板审核
  └─ 同时：
     ├─ 保存到聊天历史（可复盘、学习）
     ├─ 提取关键知识点（自动补充"学习档案"）
     ├─ 检测知识库缺口（被问但无答案的话题）
     └─ 生成日报数据（用于后台统计）
```

## 📊 数据流与权限边界

### 三层数据架构

| 层级 | 数据 | 老板能看 | 店长能看 | 咨询师/美容师能看 | 备注 |
|-----|------|---------|---------|--------------|------|
| **聊天记录** | 所有员工的提问 & AI 回答 | ✅ | ✅ 仅本店 | ❌ 仅自己 | 敏感：员工提问反映知识掌握度 |
| **客户档案** | 客户基本信息、消费历史、标签 | ✅ | ✅ 仅本店 | ✅ 仅自己跟进的客户 | 按岗位需要授权 |
| **知识库** | SOP、话术、项目介绍 | ✅ 编辑权 | ✅ 编辑权 | ✅ 只读 | 按 visible_roles 过滤 |
| **日报统计** | 每日问题汇总、能力评估 | ✅ | ✅ 仅本店 | ❌ | 用于管理决策 |
| **风险日志** | 高风险提问、违规词检测、客诉 | ✅ 全部可见/处理 | ✅ 仅本店 | ❌ | 合规底线 |
| **任务** | 分配给各角色的任务 | ✅ 创建/编辑 | ✅ 创建/编辑 | ✅ 查看和完成自己的 | 可按职位拆解 |

### 权限检查点

```
【页面加载时】
  ├─ /admin → 判断 canEnterAdmin()：只有 owner/manager 能进
  ├─ /admin/knowledge → 判断 hasPermission("knowledge", "view")：可看
  ├─ /admin/knowledge/upload → 判断 hasPermission("knowledge", "create")：可上传
  └─ /admin/risks → 判断 hasPermission("risks", "handle_risk")：可确认

【API 调用时】（lib/actions.ts 中）
  ├─ uploadKnowledge() → requireAdmin() + hasPermission("knowledge", "create")
  ├─ createTask() → requireAdmin()
  ├─ confirmRisk() → hasPermission("risks", "handle_risk")
  └─ exportReport() → hasPermission("reports", "export")
```

## 🎯 典型业务场景

### 场景 1：新项目上线（周期：3-5 天）

```
Day 1: 老板/产品团队
  └─ 撰写项目资料（施术流程、禁忌、卖点、价格、话术）

Day 2: 老板在 /admin/knowledge 上传
  └─ 上传 5-10 份资料（PDF / Word）
  └─ 标记为"项目介绍"、"护理流程"、"销售话术"分类
  └─ 选择可见角色（咨询师、美容师、前台）
  └─ AI 自动切分、向量化、集成到检索库

Day 3: 店长给员工培训
  └─ 在 /admin/tasks 创建"项目培训"任务分配给全员
  └─ 或在晨会上演示，让员工自己在 /chat 问 AI 相关问题

Day 4: 员工使用
  └─ 咨询师问 "怎么介绍这个项目"
  └─ 美容师问 "这个项目的标准流程"
  └─ 前台问 "价格表和活动折扣"
  └─ AI 返回基于新资料的精准答案

Day 5: 复盘数据
  └─ 老板看 /admin/reports 查看"项目提问热度"
  └─ 识别员工的知识掌握情况和话术误区
  └─ 补充更细致的 SOP 或反面案例
```

### 场景 2：客诉升级处理（周期：实时 → 1 小时内）

```
T+0min: 前台/咨询师遇到客户投诉
  └─ "做完项目泛红 3 天还没消，客户要求退费"
  └─ 在 /chat 问 AI "这种情况怎么处理"

T+1min: AI 处理流程
  ├─ 检索知识库"客诉处理"分类
  ├─ 发现有"泛红"的标准解释 / 处理流程
  ├─ 生成员工可用的回复：安抚方案、不当承诺检测等
  └─ 但因涉及退费/赔偿，自动评为 L3 或 L4（需老板确认）

T+5min: 系统升级给老板
  └─ 在 /admin/pending 上出现"待确认的客诉回答"
  └─ 老板看到完整的客诉背景、AI 建议、可用的标准话术
  └─ 老板做最终决策：同意退费 / 优惠 / 进一步协商

T+30min: 老板返回建议
  └─ 在系统中回复员工的确认
  └─ 同时：自动提取经验到"客诉处理"知识库
     （下次类似客诉时，员工能直接问到标准方案）

T+60min: 客户满意解决，流程闭环
```

## 🛡️ 风险管理流程

### 风险等级定义

| 等级 | 名称 | 标准 | 处理路径 | 示例 |
|-----|------|------|--------|------|
| **L1** | 知识库明确 | 问题的答案在知识库中有完整、明确的表述 | 直接返回给员工，无需确认 | "金牌项目的价格是多少" → 知识库有标准价格表 |
| **L2** | 通用建议 | 没有知识库对应资料，但属于通用、安全的工作建议 | 直接返回给员工，标记为"通用建议" | "给客户发微信的黄金时间是几点" → AI 给出通用营销建议 |
| **L3** | 需确认 | 涉及金钱、服务承诺、权限决策的问题 | 返回给员工（可用）+ 同步给老板（作为 L3 待确认） | "客户只交 50% 定金可以吗" / "给老客户额外优惠 500" |
| **L4** | 高风险 | 医美违规表达、医学声称、违禁词、严重违承 | 阻止返回，直接升级给老板，员工看不到 | "这个项目根治痘印" / "保证一次见效" / "排毒" |

### 风险检测机制

```
【1. 禁用词检测】
  输入：AI 生成的回答文本
  检查：是否包含门店配置的禁用词表
  禁用词来源：
    ├─ 系统默认 15 个医美违规词（根治、永久、保证、排毒等）
    ├─ 老板在 /admin/knowledge → 禁用词 手动补充的词
    └─ AI 自动学习（从历史高风险提问中提取新词）
  返回：如果命中任何禁用词 → L4
  
【2. 医美合规检测】
  输入：AI 生成的回答文本
  检查：调用医美语言合规分类器（classification model）
  判断标准：
    ├─ 是否有夸大其词的医学承诺（"100% 有效"）
    ├─ 是否混淆了医学事实（把护理当成治疗）
    ├─ 是否有医学声称但缺少限定条件
    └─ 是否忽视了禁忌人群
  返回：如果检测为"合规风险高" → L4

【3. 承诺度检测】
  输入：AI 生成的回答文本
  检查：员工是否做了"过度承诺"
  判断标准：
    ├─ 明确承诺时间（"3 天见效"）
    ├─ 明确承诺效果（"绝对满意" / "不然全退"）
    ├─ 超出门店权限（"我可以给你最低价"）
    └─ 没有风险提示（施术有禁忌人群但未说明）
  返回：如果检测为"承诺过度" → L3（需老板确认）

【4. 信息完整度检测】
  输入：AI 生成的回答文本
  检查：关键信息是否完整
  判断标准：
    ├─ 如果问的是医美项目，有没有说"禁忌人群"
    ├─ 如果问的是客诉处理，有没有说"升级条件"
    ├─ 如果问的是定价，有没有说"活动限制"
    └─ 如果问的是承诺，有没有说"风险免责"
  返回：如果缺关键信息 → L3（不能直接用，需补充）
```

### 高风险流程

```
【员工提问】
  ↓
【AI 生成回答】
  ↓
【风险检测】命中 L4
  ↓
【系统行为】
  ├─ 保存聊天记录到 risk_logs
  ├─ 标记为 status="pending"（待审核）
  ├─ 在 /admin/pending 界面出现一条"高风险提问"卡片
  └─ 老板能看到完整的：员工身份、提问内容、AI 建议、命中的风险词、风险类型
  ↓
【老板行动】
  ├─ 点开查看完整对话
  ├─ 选择action：
  │  ├─ "同意"：说明这个回答是合理的（可能是误判）
  │  │    └─ 系统记录：下次 AI 碰到类似问题可以参考这次的老板决策
  │  ├─ "修改"：给出老板版的标准回答，替换 AI 的建议
  │  │    └─ 系统学习：把这个标准回答添加到知识库"标准答案"分类
  │  └─ "拒绝"：阻止员工使用这个回答
  │       └─ 系统记录：员工看不到 AI 建议，需要等老板另外指导
  ↓
【员工看到结果】
  ├─ 如果老板"同意" → 员工可以按照 AI 回答使用
  ├─ 如果老板"修改" → 员工看到老板给的标准答案
  └─ 如果老板"拒绝" → 员工看到"需要管理员确认"，等待进一步指导
  ↓
【数据沉淀】
  ├─ 记录这个风险案例到 risk_logs（用于持续改进）
  ├─ 如果频繁被问 → 补充知识库（让 AI 下次更聪明）
  ├─ 如果频繁高风险 → 增加禁用词或调整 prompt
  └─ 老板在 /admin/reports 看到"L4 风险趋势"图表
     （L4 数字 ↑ 说明需要补充培训或知识库 review）
```

## 📈 数据 & 报告体系

### 日报数据（每日自动生成）

```
【统计维度】
  ├─ 提问总数：今日全店员工提问数
  ├─ 知识库命中率：有明确答案的提问占比（L1）
  ├─ 平均置信度：检索的相关度评分
  ├─ 问题分类分布：各类型问题的占比
  ├─ 风险提问数：L3 + L4 的数量
  ├─ 禁用词命中：违规表达的检测次数
  ├─ 客户新增/跟进/成交：对接客户系统（可选）
  └─ 任务完成率：今日完成的任务数 / 分配任务数

【角色维度的日报】
  ├─ 咨询师：提问数、销售话术相关提问 % 、客户跟进数
  ├─ 美容师：提问数、项目流程相关提问 % 、服务投诉数
  ├─ 前台：提问数、接待/预约相关提问 % 、新客咨询数
  ├─ 运营：提问数、文案生成相关提问 % 、发布数量
  ├─ 店长：巡店反馈数、决策数、审核高风险数
  └─ 老板：（综合视图）

【知识库维度的日报】
  ├─ 被问最频繁的 TOP 5 话题
  ├─ 知识库缺口（被问但无答案的）TOP 5
  ├─ 最近一周新增资料数量
  └─ 禁用词命中榜
```

### 周报 / 月报（汇总）

```
【经营指标】
  ├─ 员工工作效率：提问数 ÷ 工作时间（高频提问说明知识掌握度不足）
  ├─ AI 辅助度：L1 比例（知识库完整度指标）
  ├─ 风险管理：L4 数量、处理时间、老板确认率
  ├─ 任务执行率：完成 / 逾期 / 取消的比例
  └─ 客户效果（如集成）：新客咨询数 → 到店数 → 成交数 → 复购数

【员工能力评估】（自动生成，基于大模型的语义分析）
  ├─ 咨询师 A：销售话术掌握度 85% / 客户异议处理 72%（缺口：高端产品介绍）
  ├─ 美容师 B：项目流程标准化 92% / 应急处理 65%（缺口：过敏应对）
  ├─ 前台 C：接待规范 88% / 预约管理 79%（缺口：活动推介）
  └─ ...

【知识库优化方向】
  ├─ 需要补充的话题（缺口 TOP 10）
  ├─ 需要改进的文档（歧义、不清楚、冗长）
  ├─ 需要增加的禁用词（从风险提问中学习）
  ├─ 最受欢迎的资料（被检索频次最高）
  └─ 建议优化的旧资料

【合规 & 风险】
  ├─ L4 高风险数量趋势（↑↓↔）
  ├─ 违禁词排行榜（哪个词最常被误用）
  ├─ 老板审核响应时间（平均多久处理 L4）
  ├─ 错误学习率（AI 被纠正的比例）
  └─ 重复客诉（同类客诉频次）

【人力资源建议】
  ├─ 哪个岗位、哪个员工需要重点培训
  ├─ 建议的培训内容（基于缺口数据）
  ├─ 能力突出的员工（可作为讲师）
  └─ 绩效奖励的数据支撑（提问数低 + 命中率高 = 知识掌握好）
```

## 🔧 技术架构（面向业务的简化版）

### 可替换适配层

```
页面/组件 (React)
    ↓
lib/actions.ts (Server Actions)
    ↓
    ├─ lib/auth/ ← 登录态（Supabase 或其他认证服务）
    ├─ lib/db/ ← 数据访问（Supabase 或 MySQL/国内云数据库）
    ├─ lib/storage/ ← 文件存储（Supabase / MinIO / 阿里云 OSS）
    └─ lib/ai/ ← AI 模型调用（mock / deepseek / openai / claude）
        ↓
        下层实现细节（可替换，不影响业务流程）
```

**快速切换 AI 模型**：
```bash
# .env.local
AI_PROVIDER=mock        # 本地演示（无需 API Key）
AI_PROVIDER=deepseek    # 国内推荐
AI_PROVIDER=openai      # 海外推荐
AI_PROVIDER=claude      # Anthropic（推荐用于医美合规）
```
